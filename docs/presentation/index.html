Why timetabling? Every university has to solve this problem every semester — assigning courses to rooms and time slots. It seems simple, but once you have hundreds of courses, multiple rooms of different sizes, and constraints from curricula, teachers, and room availability, it becomes an NP-hard combinatorial optimization problem. That makes it a perfect candidate for a knowledge-based approach.

Let me walk through the problem structure. Each instance file defines courses with a teacher, number of lectures, and enrollment count. Rooms have capacities. Curricula group courses that share students — these can't overlap. And there are unavailability constraints — specific time slots where a course cannot be scheduled. In our Prolog system, we represent these using compound terms — course/5, room/2, curriculum/2 — and bundle everything into a Prolog dict called Instance. The solution is a list of assignment/5 terms, each mapping a lecture of a course to a day, period, and room.

Hard Constraints

Feasibility = ALL Must Hold

✅ feasible(I, S)
AND — all 5 required
Lecture Coverage
Every course gets exactly the required # of lectures
No Room Conflicts
≤1 course per (Day, Period, Room)
No Teacher Conflicts
A teacher can't teach two courses at the same time
No Curriculum Conflicts
Courses in the same curriculum can't overlap
No Unavailability Violations
Course not placed in forbidden (Day, Period)

feasible(Instance, Solution) :-
    \+ violates(Instance, Solution, _).

violates(Instance, Solution, Reason) :-
    ( violates_lecture_count(Instance, Solution, Reason)
    ; violates_room_conflict(Solution, Reason)
    ; violates_teacher_conflict(Instance, Solution, Reason)
    ; violates_curriculum_conflict(Instance, Solution, Reason)
    ; violates_unavailability(Instance, Solution, Reason)
    ), !.

%% Example: room conflict — same room, same slot
violates_room_conflict(Sol, room_conflict(D,P,R)) :-
    member(assignment(C1, _, D, P, R), Sol),
    member(assignment(C2, _, D, P, R), Sol),
    C1 \= C2.
A solution is feasible only if ALL five hard constraints are satisfied — this is an AND relationship, as shown in the diagram. Lecture Coverage — every course must get exactly the specified number of lectures. No Room Conflicts — two courses can't share a room at the same time. No Teacher Conflicts — same teacher can't teach two courses in the same slot. No Curriculum Conflicts — courses sharing students can't overlap. No Unavailability — respect forbidden time slots. In the code, feasible/2 simply checks that no violation exists. The violates/3 uses disjunction — if any single constraint is broken, the solution is infeasible.

Soft Constraints

Penalty Minimization

Total Penalty = Σ all penalties
+   +   +   +
🏫
Room Capacity
Weight: ×1
students − capacity
per over-capacity lecture
e.g. 130 in 100-seat room → 30
📅
Min Working Days
Weight: ×5
(minDays − actualDays) × 5
per under-spread course
e.g. needs 4, got 3 → 5 penalty
🔗
Curriculum Compactness
Weight: ×2
+2 per isolated lecture
no adjacent curriculum lecture
encourages grouping
🚪
Room Stability
Weight: ×1
(numRooms − 1)
per course using multiple rooms
e.g. 3 rooms → 2 penalty

penalty(Instance, Solution, TotalPenalty) :-
    room_capacity_penalty(Instance, Solution, RoomP),
    min_working_days_penalty(Instance, Solution, DaysP),
    curriculum_compactness_penalty(Instance, Solution, CompactP),
    room_stability_penalty(Instance, Solution, StableP),
    TotalPenalty is RoomP + DaysP + CompactP + StableP.
Once a solution is feasible, we want to minimize the penalty. ITC2007 defines four soft constraints, each with a different weight. Room Capacity — 130 students in a 100-seat room costs 30, weight 1. Minimum Working Days — lectures should be spread across distinct days; missing a day costs 5. Curriculum Compactness — isolated lectures in a curriculum cost 2 each. Room Stability — each extra room used by a course adds 1. The total penalty is the sum. Our greedy solver for comp01 achieves about 3,020.

Solver 1

Greedy Constructive Approach

Step 1
Build Tasks
One task per lecture (6 lectures → 6 tasks)
▼
Step 2
Rank by Difficulty
More constraints → schedule first
▼
Step 3
Greedy Placement
Try (slot × room) until feasible
▼
Step 4
Retry if Stuck
Re-randomize, up to 30 attempts

construct(Instance, Solution) :-
    build_tasks(Instance, CurrMap, Tasks),
    try_attempts(30, Instance, Rooms, Slots,
                 TeacherMap, CurrMap, Tasks, Solution).

% Difficulty: higher weight = schedule first
task_weight(Instance, CurrMap, task(Course, _), W) :-
    findall(1, member(unavailable(Course,_,_),
            Instance.unavailability), Unav),
    length(Unav, UnavCount),
    W is UnavCount + CurrCount.

% Constraint checks during placement
can_place(Course, Day, Period, RoomId, Instance, ...) :-
    \+ memberchk(unavailable(Course,Day,Period), ...),
    \+ memberchk(assignment(_,_,Day,Period,RoomId), Acc),
    %% + teacher & curriculum conflict checks
Our first solver is a greedy constructive approach. First, we create one task per lecture. Then we rank tasks by difficulty — courses with more unavailability constraints or curriculum overlaps are scheduled first, because they have fewer valid slots. We generate all possible time slots, randomize them, and for each task try every slot and room until we find one that passes all hard constraints. If we get stuck, we retry with a new random permutation — up to 30 attempts. This approach is fast — comp01 solves in under a second — and successfully finds feasible solutions for 20 out of 21 instances. The one failure, comp05, has 771 unavailability constraints.

Solver 2

CLP(FD) Constraint Solver

Phase 1: Model

One variable per lecture:

  • Slot[i] ∈ 0..NumSlots-1
  • Room[i] ∈ 1..NumRooms

Domains encode all possible assignments

Phase 2: Post Constraints

  • Unavailability: Slot[i] #\= Forbidden
  • Same course: Slot[i] #\= Slot[j]
  • Same teacher: Slot[i] #\= Slot[j]
  • Same curriculum: Slot[i] #\= Slot[j]
  • Room conflict:
    (Slot[i] #\= Slot[j]) #\/ (Room[i] #\= Room[j])

Phase 3: Search

labeling([ffc], Vars)

ffc = First-Fail, most Constrained

Smallest domain first


construct(Instance, Solution) :-
    build_lectures(Instance.courses, Lectures),
    length(Lectures, N),
    length(SlotVars, N),  length(RoomVars, N),
    SlotVars ins 0..MaxSlot,  RoomVars ins 1..NumRooms,
    constrain_unavailability(Lectures, SlotVars, ...),
    constrain_pairs(Lectures, SlotVars, RoomVars, ...),
    %% Room: (Slot1 #\= Slot2) #\/ (Room1 #\= Room2)
    labeling([ffc], Vars),
    build_solution(Lectures, SlotVars, RoomVars, ..., Solution).
Our second solver uses CLP(FD) — Constraint Logic Programming over Finite Domains, built into SWI-Prolog. Instead of placing lectures one by one, we create variables — one slot and one room variable per lecture, with finite domains. Then we post all hard constraints as mathematical constraints. For example, same-teacher courses get Slot[i] not-equal Slot[j]. The room conflict says: either different slots OR different rooms — a reified disjunction. Finally, labeling with the ffc heuristic searches the solution space. CLP(FD) is elegant but computationally heavier — it solves 5 of 21 instances within timeout. But it guarantees all hard constraints by construction.

🧠 Expert System Pattern

Knowledge Base (constraint rules) separated from Inference Engine (solvers). Adding a new constraint = adding a rule, not rewriting the solver.

✅ Key Takeaways

Prolog's pattern matching → natural constraint checking.
SWI-Prolog dicts → readable structured data.
CLP(FD) → powerful but search space explodes on large instances.

🔮 Future Improvements

Local search (simulated annealing) to reduce penalty.
Symmetry breaking in CLP(FD) model.
Hybrid: CLP(FD) for small, greedy for large instances.

Here's our architecture. The key design decision was separating the knowledge base — constraint rules — from the solvers. This is a classic expert system pattern. The rules define WHAT must hold, the solver decides HOW to satisfy them. Prolog was a great fit. Pattern matching makes constraint checking natural. Dicts make the model readable. The main lesson: CLP(FD) is elegant and correct by construction, but greedy is far more practical for large instances. Future work would add local search to optimize penalty.
